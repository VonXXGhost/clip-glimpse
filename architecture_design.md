# ClipGlimpse — Architecture Design

## 1. 需求概述

### 问题场景
- 云电脑只允许本地→云端单向传输（文件/剪贴板），云端→本地被封禁。
- 需要一种**合规、离线、低系统权限**的方式，将云电脑上的文字内容回传到本地 PC。

### 解决方案
- 云端运行 **生成模式**：将文字分片编码为二维码，循环动态显示。
- 本地运行 **读取模式**：通过快捷键触发屏幕区域监视，逐片解码二维码，重组文本并存入历史面板。

### 约束
1. **企业合规**：不依赖任何第三方商业网络服务，纯本地操作。
2. **仅 Windows**：目前只需支持 Windows 运行。
3. **低资源占用**：运行时 CPU/内存开销要小。
4. **低系统权限**：不需要管理员权限，不需要注入/钩子等高敏感操作。

## 3. 数据传输协议 (Wire Protocol v2)

### 3.1 分片格式

每个二维码携带一个 `chunk`，由 12 字节固定头 + 经 lz4 压缩的 payload 组成。

所有分片格式统一，不再区分 SOS/DATA/EOS：

```
┌────────┬──────────┬─────────┬─────────┬──────────┬──────────────────┐
│  MAGIC │ VERSION  │  FLAGS  │   SEQ   │  TOTAL   │      CRC32      │
│  2 B   │  1 B     │  1 B    │  2 B    │  2 B     │      4 B        │
│  "CG"  │  0x02    │bit0=lz4 │ u16 BE  │ u16 BE   │    u32 BE       │
└────────┴──────────┴─────────┴─────────┴──────────┴──────────────────┘
┌──────────────────────────────────────────────────────────────────────┐
│                       PAYLOAD (N 字节)                                │
│                    lz4 压缩后的 UTF-8 文本分片                          │
└──────────────────────────────────────────────────────────────────────┘
```

| 字段     | 大小 | 说明 |
|---------|------|------|
| MAGIC   | 2B   | 固定为 `0x43 0x47`（ASCII "CG"），用作二维码内容识别 |
| VERSION | 1B   | 协议版本，当前为 `0x02` |
| FLAGS   | 1B   | bit0=1 表示 payload 经 lz4 压缩；bit1 预留（彩色模式标记） |
| SEQ     | 2B   | 分片序号，从 0 开始到 total-1 |
| TOTAL   | 2B   | 总分片数（最大 100） |
| CRC32   | 4B   | **完整原始文本**的 CRC32 校验值，所有分片相同 |
| PAYLOAD | N B  | lz4 压缩后的分片数据（解压后为 UTF-8 文本段） |

### 3.2 QR 码参数选择

QR 码的版本、纠错级别、模块像素大小是吞吐量的核心变量，三者互相制约：

**版本与容量对照（Byte mode，原始 QR 容量）**：

| 版本 | Modules | L (~7%) | M (~15%) | Q (~25%) | H (~30%) |
|------|---------|---------|----------|----------|----------|
| 10 | 57×57 | 263 B | 211 B | 163 B | 120 B |
| 15 | 73×73 | 436 B | 353 B | 272 B | 207 B |
| 20 | 97×97 | 658 B | 541 B | 427 B | 333 B |
| 25 | 117×117 | 947 B | 779 B | 618 B | 486 B |
| 30 | 137×137 | 1264 B | 1043 B | 832 B | 660 B |
| 35 | 157×157 | 1603 B | 1330 B | 1064 B | 847 B |
| 40 | 177×177 | 1948 B | 1624 B | 1304 B | 1045 B |

**模块像素与 QR 渲染尺寸**（每个 module 映射到屏幕像素）：

| px/module | V25 尺寸 | V30 尺寸 | 可读性 |
|-----------|----------|----------|--------|
| 1px | 117×117 | 137×137 | 不可靠，远程桌面压缩后模糊 |
| 2px | 234×234 | 274×274 | 通常可用，压缩轻微时 OK |
| 3px | 351×351 | 411×411 | 安全，即使压缩/缩放也清晰 |
| 4px | 468×468 | 548×548 | 非常安全，但占用屏幕大 |

**推荐预设方案**（Payload 列为原始 QR 容量，减去 12B 协议头后为有效负载）：

| 预设 | 版本 | EC | px/mod | 原始容量 | 有效负载 | 窗口大小 |
|------|------|-----|--------|---------|---------|----------|
| 保守 | V20 | Q | 3px | 427 B | 415 B | 291×291 |
| **默认** | **V25** | **M** | 3px | **779 B** | **767 B** | 351×351 |
| 高速 | V30 | M | 2px | 1043 B | 1031 B | 274×274 |
| 极限 | V35 | L | 2px | 1595 B | 1583 B | 314×314 |

### 3.3 吞吐量估算

假设循环间隔 300ms（即每秒显示约 3.3 个 QR），数据经过 lz4 压缩，实际传输大小因文本可压缩性而异，下表按原始有效负载估算：

| 文本长度 | 默认(V25/M/3px, 767B/片) | 高速(V30/M/2px, 1031B/片) | 极限(V35/L/2px, 1583B/片) |
|----------|-------------------------|---------------------------|--------------------------|
| 1 KB | ~2 片, 0.6s | ~1 片, 0.3s | ~1 片, 0.3s |
| 10 KB | ~14 片, 4.2s | ~10 片, 3.0s | ~7 片, 2.1s |
| 100 KB | ~134 片, 40s | ~100 片, 30s | ~65 片, 20s |
| 500 KB | ~668 片, 3m20s | ~497 片, 2m29s | ~324 片, 1m37s |
| 1 MB | ~1366 片, 6m50s | ~1017 片, 5m5s | ~663 片, 3m19s |

> **结论**：几百 KB 以内的文本（这已经是几十万汉字的量级）在几十秒内传完，对于「偶尔粘一段文字回本地」的使用场景足够实用。真正需要传 MB 级大文本时，建议分多次传输或考虑其他方式。

### 3.4 彩色 QR 模式

#### 原理

将 3 个独立的标准 B&W QR 码通过 RGB 三通道合成到一个彩色图像中。每个像素只用极值（0 或 255），共 8 种颜色：

| 颜色 | R | G | B |
|------|---|---|---|
| 黑 | 0 | 0 | 0 |
| 红 | 255 | 0 | 0 |
| 绿 | 0 | 255 | 0 |
| 蓝 | 0 | 0 | 255 |
| 黄 | 255 | 255 | 0 |
| 品红 | 255 | 0 | 255 |
| 青 | 0 | 255 | 255 |
| 白 | 255 | 255 | 255 |

#### 带宽对比 (V25-M)

| 模式 | 每帧负载 | 循环时间 (100 chunks) |
|------|----------|----------------------|
| B&W | 767 B | 100 帧 × 500ms = 50s |
| 彩色 | 2301 B (3 × 767) | 34 帧 × 500ms = 17s |

#### 实现方式

- **生成端** (`qr_gen.rs::generate_color_qr`)：对 1~3 个数据块分别调用 `QrCode::with_version`，按模块位置合成：
  - 通道 i 的 Dark → 0，Light → 255
  - 空通道全部为 255（白色）
- **读取端** (`qr_read.rs::extract_channel_from_bgra`)：从 BGRA 捕获中单独提取 R/G/B 通道，分别用标准 `rxing::decode_qr` 解码
- **对比度拉伸** (`qr_read.rs::stretch_contrast`)：通道提取后的灰度值可能因屏幕压缩导致动态范围不足（如黑字不黑、白字不白），线性拉伸 `(v - min) * 255 / (max - min)` 恢复完整灰度区间后再解码
- **不影响协议**：传输协议 (v2) 完全不变，每个 chunk 结构和编码方式一致
- **配置**：通过 `color_mode` 配置项或生成界面复选框切换，B&W 模式为默认

### 3.5 循环显示策略

```
frame[0] → frame[1] → ... → frame[N-1] → frame[0] → ...   (无限循环)
```

- 显示间隔：默认 500ms/片，可配置
- 读取端通过 `(SEQ, TOTAL)` 去重
- 变长文本时自动切换：短文本（≤1 帧）→ 静态展示，不循环；长文本（>1 帧）→ 自动启动循环

## 5. 关键技术选型

### 5.1 依赖 crate

| 用途 | crate | 理由 |
|------|-------|------|
| GUI 框架 | `eframe` / `egui` | 即时模式 UI，轻量，纯 Rust，Windows 开箱即用 |
| QR 生成 | `qrcode` | 纯 Rust 实现，成熟稳定，支持多种输出格式 |
| QR 解码 | `rxing` | ZXing 的 Rust 移植，功能全面，支持从像素缓冲区解码 |
| 屏幕捕获 | Windows GDI (via `windows` crate) | 零额外依赖，最低资源占用，无需管理员权限 |
| 系统托盘 | `tray-icon` | 纯 Rust 跨平台托盘库，API 简洁 |
| 全局快捷键 | `windows` crate (GetAsyncKeyState 轮询) | 每 50ms 轮询一次，边沿触发避免重复，无需消息循环 |
| 图像处理 | `image` | 屏幕捕获像素格式转换（BGRA→灰度）供 rxing 使用 |
| CLI 解析 | `clap` | 标准的 Rust CLI 库 |

### 5.2 备选方案与取舍

| 对比项 | 推荐方案 | 备选方案 | 说明 |
|--------|---------|---------|------|
| GUI | eframe/egui | win32 API 裸写 / iced | egui 在简单 UI + 低资源场景下最平衡 |
| 屏幕捕获 | windows GDI BitBlt | scrap / xcap / screenshots | BitBlt 无需额外 DLL，性能可控 |
| QR 解码 | rxing | rqrr | rxing 对屏幕拍摄/截图的鲁棒性更好 |
| 图像格式 | image crate | 自制转换 | image crate 功能齐全，编译后体积可接受 |

### 5.3 关于低系统权限
- **不需要管理员权限**：普通用户即可运行
- **不需要 hook/inject**：纯屏幕捕获 + 图像识别
- **不需要网络访问**：完全离线
- **不需要注册系统服务**：以普通应用程序运行
- **注册快捷键**：通过 `GetAsyncKeyState` 每 50ms 轮询，边沿触发避免重复触发

## 7. 核心算法流程

### 7.1 生成流程

```
用户输入文本
  ↓
程序自动检测文本输入更新
  ↓
全文 lz4 压缩 → 按 max_payload 切分 → 构造均匀 chunk（无 SOS/DATA/EOS）
  ↓
每 chunk 携带相同 CRC32（全文校验）
  ↓
  ├─ B&W 模式: 每 chunk 调用 QrCode::new(bytes) → 单帧单 QR
  └─ 彩色模式: 每 3 chunk 一组 → generate_color_qr → RGB 三通道合成单帧
  ↓
  ├─ 帧数 ≤ 1 → 静态展示，不循环
  └─ 帧数 > 1 → 按 interval 在 UI 上循环显示
```

### 7.2 读取/重组流程

```
按下热键 → 启动扫描
  ↓
每 scan_interval (例如200ms) 执行:
  ↓
截取屏幕指定区域的像素 (BitBlt → BGRA buffer)
  ↓
  ├─ B&W 模式: 转为灰度图 (0.299*R + 0.587*G + 0.114*B) → 解码
  └─ 彩色模式: 依次提取 R/G/B 通道
       ├─ 对比度拉伸 (stretch_contrast) → 恢复动态范围
       └─ 各通道分别解码
  ↓
rxing::BarcodeReader 尝试解码
  ↓
  ├─ 解码失败 → 继续下一帧/通道
  └─ 解码成功 → 解析 chunk
       ↓
      检查 magic: 不是 "CG" → 丢弃
       ↓
      检查 version: 不是 0x02 → 丢弃
       ↓
      按 (SEQ, TOTAL) 去重
       ↓
      buffer[seq] = payload
       ↓
      收齐 total 个不重复 chunk → lz4 解压 → CRC32 校验
       ↓
      校验通过 → 存入 history → 自动写入剪贴板 → Toast 通知 → 停止扫描
      CRC 失败 → 丢弃，继续等待
       ↓
      继续下一帧扫描
```

### 7.3 组装完成判断

```
条件: buffer[0..total] 全部非空（所有分片到齐）
触发: 每写入一个新 chunk 检查一次满仓
超时: 收到第一个 chunk 后 30 秒内未完成 → 重置 assembler
去重: 同一 chunk 的多次读数通过 (SEQ+TOTAL) 忽略
校验: lz4 解压 → CRC32 与各 chunk 携带的全文 CRC32 比对 → 一致才交付
```

## 8. 实现阶段规划

### Phase 1: 核心（协议 + 编解码）
1. `protocol.rs` — struct ChunkHeader, encode/decode 函数, 单元测试
2. `qr_gen.rs` — QR 生成（输入 bytes → 输出 image::RgbImage）
3. `qr_read.rs` — QR 解码（输入灰度像素 → 输出 bytes）
4. 手动测试：生成 → 屏幕截图保存为图片 → 解码 → 验证往返

### Phase 2: 生成模式 GUI
1. `generate/ui.rs` — eframe 窗口，文本框 + 图片渲染 + 循环控制
2. 启动参数通过 `cli.rs` (子命令 `generate`)

### Phase 3: 读取模式 GUI + 扫描
1. `screen.rs` — 屏幕区域截取
2. `read/scanner.rs` — 扫描循环
3. `read/ui.rs` — 历史面板 + 选区设置
4. `tray.rs` + `hotkey.rs` — 系统托盘 + 快捷键
5. `history.rs` — 内存历史存储

### Phase 4: 打磨
1. 选区交互优化（拖拽选区 overlay）
2. QR 显示大小自适应窗口
3. 播放速度和分片大小预设
4. 错误提示与边界情况处理
5. 低 CPU 模式下降低扫描帧率

## 9. 技术风险与缓解

| 风险 | 概率 | 缓解措施 |
|------|------|---------|
| 远程桌面压缩导致 QR 不可读 | 中 | 使用更高纠错级别（Q/H），增大 QR 显示尺寸 |
| 扫描帧率不够导致漏片 | 低 | 循环显示（错过可等下一轮），160ms 扫描间隔即可 |
| rxing 对屏幕截图解码失败率高 | 中 | 测试多种图像预处理：提高对比度、锐化、二值化 |
| 不同 DPI/缩放导致选区偏移 | 中 | 使用 DPI-aware API，选区坐标用物理像素 |
| egui 纹理刷新耗时 | 低 | 限制 QR 渲染帧率，用纹理缓存 |

## 10. 开放问题（需要决策）

1. **单二进制还是双二进制？** 建议 `clip_glimpse generate` 和 `clip_glimpse read` 两个子命令，同一个 exe，用 `clap` 区分。用户决策：同意。

2. **QR 大小是否自动适应窗口？** 建议固定 QR module 像素大小（如 3px/module），整体随版本变化；窗口大小跟随 QR。用户决策：同意。

3. **是否需要支持多显示器？** v1 只支持主显示器，后续可扩展。用户决策：不需要支持。

4. **配置持久化？** 选区坐标、快捷键、扫描间隔等配置可保存到 `%APPDATA%/clip_glimpse/config.toml`（本地文件，不上传，合规）。用户决策：同意配置持久化，但目录使用 `./config.toml`。

5. **是否需要 Toast 通知？** 建议用 `notify-rust` 或 WinRT Toast API，收到新文本时弹出通知。用户决策：同意。

6. **最大传输长度？** 理论无限（取决于分片数量），但建议 UI 提示总时间和分片数估算。2B seq 支持 65535 片，每片 477B → 最大约 31MB。用户决策：最大分片数量可配置，默认为100，超出最大值后在生成端进行提示。

## 11. 替代方案对比

| 方案 | 优点 | 缺点 | 
|------|------|------|
| **QR 码传输（本项目）** | 零网络，合规，通用 | 需要两个设备屏幕互通，速度慢 |
| 本地 HTTP 服务 + 剪贴板 | 速度快 | 需要网络通路（云电脑禁了回传方向） |
| 虚拟声卡/音频编码 | 隐蔽 | 实现复杂，云电脑可能禁音频设备 |
| OCR 方案（直接扫描文字） | 无需特殊客户端 | 不准确，无法处理格式化文本 |
| 基于文件共享的同步盘 | 方便 | 云电脑通常禁止安装第三方同步盘 |

结论：QR 码方案是满足合规约束下的最可行方案。

## 12. 最小可行产品 (MVP) 的简化路径

MVP 可以这样快速验证：

**生成端**：直接用 Python/Node.js 脚本生成 QR 码，用浏览器打开一个 HTML 页面循环显示（验证阶段可用，最终用 Rust 重写）。

**读取端**：核心 Rust 程序，只做：
1. 无 GUI 命令行模式（`--capture` 指定区域坐标和持续时间）
2. 截屏 → 解码 → 打印到 stdout
3. 验证端到端可行性后，再加 GUI

这样分步骤走，降低整体实现风险。
